

## Goal
Open the Team Briefing (currently CEO-only) for read access to every authenticated Kabuni user, while keeping **generation/regeneration** restricted to the CEO (Nimesh).

## Current Behaviour
- `/ceo` route + `CEOBriefing` page are gated by `isCEO(email)` against an allowlist (`nimesh@kabuni.com`, `palash@kabuni.com`) in `src/lib/ceoAccess.ts`.
- `useCEOBriefing` reads from `ceo_briefings` and invokes `ceo-briefing` (generate) / `ceo-briefing-status` (poll) edge functions.
- RLS on `ceo_briefings` and the edge functions both check `CEO_EMAILS` for read + write.
- Sidebar entry for the briefing is hidden from non-CEOs.

## Changes

### 1. Access model
Introduce two roles in code:
- **Viewer**: any authenticated `@kabuni.com` user → can open `/ceo`, read latest + previous briefings, see all panels (TLDR, Risk Radar, Workstreams, etc.).
- **Generator**: CEO only (`nimesh@kabuni.com`) → can click **Regenerate** and trigger the pipeline.

Add `CEO_GENERATOR_EMAILS = ["nimesh@kabuni.com"]` and helper `canGenerateBriefing(email)` in `src/lib/ceoAccess.ts`. Keep `isCEO` as the viewer check but broaden its semantics (or add `canViewBriefing` returning true for any signed-in user — preferred, cleaner).

### 2. Frontend
- `src/pages/CEOBriefing.tsx`: remove the `isCEO` redirect. Allow any authenticated user. Conditionally render the **Regenerate** button + job progress UI only when `canGenerateBriefing(user.email)`. Non-generators see a subtle "Last generated by CEO at …" caption instead.
- `src/components/Sidebar.tsx` (and any nav surfacing the entry): show the "Team Briefing" link for everyone, drop the CEO-only guard.
- Rename user-facing label from "CEO Briefing" → "Team Briefing" wherever it appears in the UI (sidebar, page header, page title). Internal route `/ceo` and table names stay unchanged to avoid churn.
- `useCEOBriefing`: leave `load()` untouched (read works for everyone once RLS is updated). Keep `generate()` as-is; it will simply fail server-side for non-generators — but we'll also hide the trigger in UI so this is defense-in-depth.

### 3. Backend (edge functions)
- `supabase/functions/ceo-briefing/index.ts`: keep the email allowlist check, but tighten it to `CEO_GENERATOR_EMAILS` (Nimesh only). Return 403 for anyone else attempting to generate.
- `supabase/functions/ceo-briefing-status/index.ts`: relax — allow any authenticated `@kabuni.com` user to poll job status (so viewers can watch a CEO-triggered run progress in real time if desired).
- `supabase/functions/send-ceo-briefing-actions/index.ts`: keep CEO-only (it sends actions on behalf of the CEO).
- `supabase/functions/norman-chat/index.ts`: leave existing CEO-mode gating untouched (separate feature).
- Update the comment block in `src/lib/ceoAccess.ts` to reflect the new split.

### 4. Database (RLS)
Migration on `ceo_briefings` table:
- **SELECT** policy: replace CEO-email-only check with `auth.uid() IS NOT NULL` (any authenticated user can read). Optionally restrict to `@kabuni.com` via `auth.jwt() ->> 'email' LIKE '%@kabuni.com'`.
- **INSERT / UPDATE / DELETE** policies: tighten to Nimesh's email only.
- Same treatment for `ceo_briefing_jobs` if it has RLS that currently mirrors the briefings table (SELECT open to authenticated users, write restricted to Nimesh / service role).

### 5. Copy & polish
- Page header subtitle for non-generators: "Read-only view. Generated daily by the CEO."
- Disable any "Send actions" / mutation buttons for non-generators (greyed with tooltip "CEO only").

## Out of Scope
- No change to the briefing generation pipeline, prompt, models, or token usage (separate ongoing work).
- No change to `/ceo` route path (URL stays the same to preserve bookmarks).
- Palash's access: under the new model he becomes a **viewer** (loses generate rights). Confirm if he should retain generate — see Open Question.
