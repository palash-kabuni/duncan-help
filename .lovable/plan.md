

## Expand Section 07 "Automation Progress" into a real productivity signal

### Today

Section 07 currently renders a thin slice of `payload.automation_progress` — usually just an automation count and a one-liner. It doesn't reflect that Duncan now tracks every AI call in `token_usage`, so the CEO has no visibility into:

- how much the company is actually leaning on Duncan
- who the power users are
- where Duncan is creating leverage vs. where it's still missing
- what to build next

### What Section 07 will become

A three-block "Automation & Leverage" section grounded in real `token_usage` + briefing signals:

**Block A — Company usage (last 30 days)**
- Total tokens used
- Total AI requests
- Active users (distinct user_ids with usage)
- Day-over-day and week-over-week trend (▲/▼ %)
- Trend interpretation: "Adoption accelerating", "Flat", "Declining"

**Block B — Top 3 power users (last 30 days)**
- Rank, name, role/department (joined from `profiles`)
- Tokens used + request count
- One-line "what they're using Duncan for" inferred from their dominant chat surfaces (general chat vs. project chats vs. CEO briefing vs. Gmail drafting). Derived deterministically from counts in `general_chat_messages`, `project_chats`, `gmail_writing_profiles.auto_drafts_created_today`, etc. — no new tables.
- Productivity signal: estimated hours saved = `total_tokens / 4 (chars) / 250 (wpm) / 60` as a rough but consistent proxy, labelled clearly as an estimate.

**Block C — Top 3 recommendations for Duncan**
Generated by the AI from a combined evidence pack:
- Coverage gaps (`payload.coverage_gaps`)
- Silent priorities (`headline_context.silent_priorities`)
- Friction items where `recommended_resolver != "CEO"` (automatable)
- Workstreams stuck Yellow/Red for >7 days
- Heaviest manual surfaces (e.g. lots of Gmail drafts → suggest auto-categorisation; lots of repeat questions in general chat → suggest a saved workflow)
- Failed/empty tool calls if surfaced in `payload`

Each recommendation has:
- `title` (e.g. "Auto-summarise weekly Drive reports for Ops")
- `why_now` (evidence: which gap / friction / heavy surface it addresses)
- `expected_leverage` (Low / Medium / High)
- `effort` (S / M / L)
- `auto_injected: boolean` + `evidence_source` (same pattern as Sections 03 / 05 / 06)

A deterministic floor guarantees at least 3 recommendations on any non-green briefing, drawn from coverage gaps + heaviest manual surfaces, even if the model returns `[]`.

### Implementation

**1. `supabase/functions/ceo-briefing/index.ts`**
- Add a `fetchAutomationLeverage()` helper that queries:
  - `token_usage` for last 30 days → company totals, trend deltas, top 3 user_ids
  - `profiles` for names/roles of those user_ids
  - lightweight counts from `general_chats`, `project_chats`, `meetings`, `gmail_writing_profiles` to classify what each top user uses Duncan for
- Inject the resulting `automation_leverage` block into the AI evidence pack
- Extend `MORNING_SCHEMA_HINT` for `payload.automation_progress`:
  ```ts
  {
    company_usage: { total_tokens, request_count, active_users,
                     dow_change_pct, wow_change_pct, trend_label },
    top_users: [{ rank, name, role, department,
                  total_tokens, request_count,
                  primary_use, est_hours_saved }],
    recommendations: [{ title, why_now, expected_leverage,
                        effort, auto_injected, evidence_source }]
  }
  ```
- Add a deterministic post-processor that fills `recommendations` from coverage gaps + silent priorities + heaviest manual surfaces if the model returns fewer than 3.

**2. `src/pages/CEOBriefing.tsx`**
- Replace the current Section 07 body with three labelled sub-blocks:
  - **Usage** — a compact stat row (Tokens / Requests / Active users / Trend)
  - **Top 3 users** — ranked rows with name, role, tokens, est. hours saved, primary use chip
  - **Recommendations** — top 3 cards with leverage + effort badges and the same `Auto-flagged` + `Source` chips already used in Sections 05/06
- Keep the existing section header and confidence/evidence styling
- Add an empty state for green briefings: "Adoption healthy. No new automation gaps detected today."

**3. No schema changes**
- All data comes from existing tables (`token_usage`, `profiles`, `general_chats`, `project_chats`, `meetings`, `gmail_writing_profiles`).
- No new RLS, no new migrations.
