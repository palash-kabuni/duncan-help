

## Fix: Leadership Performance is showing only one person

### What's actually happening

`payload.leadership_assessment` is generated by the AI freely — there's no rule forcing it to cover the **whole leadership team**. So when meeting/workstream signal is concentrated on one name (Simon, almost always), the model returns a single-entry array and `LeadershipGrid` faithfully renders one card. The component isn't broken; the prompt is under-specified and the context is under-supplied.

Three root causes, in order of impact:

1. **No team roster is injected.** The prompt has no list of *"these are the leaders you must assess."* The model assesses whoever happens to appear in the last 7 days of meetings/workstreams. Quiet weeks for Patrick/Alex/Matt/Parmy = absent from the grid.
2. **No minimum-coverage rule.** The schema says `leadership_assessment: LeaderAssessment[]` with no `minItems`. One row passes validation.
3. **Silence is invisible.** When a leader has no signal, the model omits them rather than reporting *"no signal this week — that itself is the finding."* For a CEO briefing, **absence of signal from a direct report is a red flag**, not a reason to hide them.

### The fix

**1. Inject a deterministic Leadership Roster**

Build `LEADERSHIP_ROSTER` from `team_directory` (filter: `is_leadership = true` or by seniority/role title — fall back to a hard-coded list of the 6–8 execs if the flag doesn't exist). Pass it into the prompt as a non-negotiable list:

```text
LEADERSHIP_ROSTER (you MUST return one entry per person):
- Nimesh (CEO)
- Patrick (CFO/Legal)
- Simon (Ops Director)
- Alex (CMO)
- Matt (CPO)
- Parmy (CTO)
- Palash (Head of Engineering)
- ...
```

**2. Schema enforcement**

```text
leadership_assessment: LeaderAssessment[]   // minItems = LEADERSHIP_ROSTER.length
                                            // exactly one entry per roster name
```

If the model returns fewer, the post-processor inserts a stub for the missing person with `signal_status: "silent"` so the CEO always sees the full team.

**3. New field: `signal_status`**

```text
signal_status: "active" | "low_signal" | "silent"
evidence_sources: string[]   // ["meetings", "workstreams", "azure", "xero", "releases"]
```

- `active` — appeared in ≥2 sources this week
- `low_signal` — appeared in 1 source
- `silent` — no appearance anywhere → **automatically flagged as `ceo_intervention_required: true`** with the reason *"No operational signal in 7 days from a direct report — confirm engagement."*

**4. Per-leader signal pre-aggregation (deterministic, not AI-judged)**

Before the prompt runs, the function tallies per-leader signals across `meetings.attendees`, `workstream_cards.owner_id`, `azure_work_items.assigned_to`, `xero` approver fields, and `recent_releases.author`. That tally is injected as `leader_signal_map` so the model assesses every roster member from real data, not from what it remembers.

**5. UI: render silent leaders with the right visual weight**

```text
EDIT src/components/ceo/LeadershipGrid.tsx
  - Add silent/low_signal styling: dashed border, muted background,
    "No signal this week" pill instead of risk_level
  - Show evidence_sources as small chips under the name
  - Sort: ceo_intervention_required → high risk → silent → low_signal → active
```

The CEO will see, at a glance, *"7 leaders. Simon: high signal, high risk. Patrick: silent — confirm. Alex: low signal — coverage gap."*

### Outcome

- The grid always shows the **entire leadership team**, never a single card.
- Silence becomes a finding, not a gap. A leader who was invisible this week is now flagged for CEO follow-up.
- Every assessment is grounded in a deterministic per-leader signal tally — no more "the model only saw Simon this week."
